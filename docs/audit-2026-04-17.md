## 1. Repair orphaned tool_use/tool_result pairs

**Commit**: `5a26f8a` · **Type**: bugfix · **Capability**: `tool-pair-repair`

### Symptom
OpenCode agent reported hitting Write/Bash tools and getting into a loop that emitted `"Tengo weasyprint..."` three times without ever executing the tool. Perceived as "the model is stuck".

### Root cause
When the client sent a conversation with an `assistant.tool_calls` block but no corresponding `tool` role response (interrupted turns, aborted executions, context-truncation mid-pair), `openaiToAnthropic` forwarded the resulting orphaned `tool_use` blocks to Anthropic. Anthropic with OAuth + tools puts the model in a degenerate state on orphaned pairs: it re-emits the same `tool_use` repeatedly because the conversation is internally inconsistent. From the client's perspective, this surfaces as the model echoing the same intent text without actually calling the tool.

### Fix
New module `src/transform/repair-tool-pairs.ts` (90 lines) implementing a two-pass O(n) sanitizer:

1. Walk all messages, collect `tool_use` IDs (from assistant content blocks) and `tool_use_id`s (from `tool_result` blocks).
2. Compute orphans via set difference.
3. If empty, return input by reference (zero allocation).
4. Otherwise, filter orphaned blocks out while preserving sibling text. Drop messages whose content ends up empty.

Invoked at the end of `openaiToAnthropic` (line 79), after cache-control annotation but before the final request body is built. Emits `transform.repairToolPairs` debug event with counts when repair fires.

### Tests
`__tests__/transform-repair-tool-pairs.spec.ts` — 9 tests covering: orphan `tool_use` removed, orphan `tool_result` removed, sibling text preserved, empty-content message dropped, valid pairs untouched, string content passthrough, referential-equality on no-op, telemetry payload shape, end-to-end through `openaiToAnthropic`.

- `__tests__/transform-repair-tool-pairs.spec.ts` — new
- `src/transform/openai-to-anthropic.ts` — import + call site

## 3. Defer client cancel mid `tool_use`

**Commit**: `59f60bb` · **Type**: feature · **Capability**: `deferred-stream-cancel`

### Symptom
When a client canceled a streaming response (closed the page, clicked stop), and the cancel happened mid `tool_use` content block, telemetry captured only a truncated `input_json_delta` — making it impossible to inspect what tool the model was about to call.

### Root cause
`reader.cancel()` in the old code path propagated immediately to the upstream Anthropic SSE reader, aborting the connection. Because Claude Pro/Max billing already counts emitted output tokens, aborting mid-block lost observability without saving cost.

### Fix
State machine in `streamAnthropicToOpenai`:
- Track `inToolUse: boolean` (set on `content_block_start` type=tool_use, cleared on matching `content_block_stop` by `event.index`).
- On `cancel()`, if `inToolUse` → set `pendingCancel=true`, record reason + timestamp, return without canceling upstream. The reader keeps consuming.
- `safeEnqueue` drops silently when `pendingCancel` is set but returns `true` so the outer loop stays alive.
- Per-event hook: if `pendingCancel && !inToolUse` → force-close (upstream canceled, stream closed).
- 30s hard ceiling (`PENDING_CANCEL_TIMEOUT_MS`): if the `tool_use` never closes, cancel anyway.

### Telemetry
- `stream.client_disconnect_deferred` (info) — cancel deferred while inToolUse=true
- `stream.client_disconnect_completed` (info) — tool_use closed after deferred cancel
- `stream.client_disconnect_timeout` (info) — 30s ceiling hit
- `stream.end` extended with `pendingCancelDeferred` and `toolUseWasOpen` flags

### Tests
`__tests__/transform-streaming-defer-cancel.spec.ts` — 8 tests covering: immediate cancel outside tool_use, deferred cancel resolved via matching `content_block_stop`, silent drop after defer, 30s timeout via `spyOn(Date, "now")`, stream.end flags, tool_use entry/exit tracking across multiple blocks, legacy-path regression.

## 5. Flush SSE buffer on stream end

**Commit**: `1026d1f` · **Type**: bugfix · **Capability**: `sse-buffer-flush`

### Symptom
If the upstream Anthropic SSE stream closed without a trailing `\n\n` after its last event (network truncation, proxy buffering boundary, upstream edge case), the final `message_delta` event was silently dropped. That event carries `stop_reason` and `usage.output_tokens` — corrupting telemetry and leaving the OpenAI-compatible response with wrong `finish_reason`.

### Root cause
The main parsing loop uses `buffer.split("\n")` + `buffer = lines.pop()!` to retain the (potentially incomplete) last line. On `done === true`, the code `break`s out of the loop, emits `data: [DONE]\n\n`, and calls `controller.close()`. The residual `buffer` was never processed. Additionally, `TextDecoder` with `{ stream: true }` holds back partial multi-byte UTF-8 bytes until `decoder.decode()` is called without the streaming flag.

### Fix
Extracted the per-event parsing logic into a local `processEvent(json): "break" | "continue"` closure (labeled-break → return value). Inserted a flush block after the main `while` loop, guarded by `!closed`:

```ts
if (!closed) {
  buffer += decoder.decode(); // flush held UTF-8 bytes
  if (buffer.length > 0) {
    for (const line of buffer.split("\n")) {
      if (!line.startsWith("data: ")) continue;
      if (processEvent(line.slice(6).trim()) === "break") break;
    }
  }
}
```

Cancel/timeout/defer-force-close paths skip flush (they set `closed = true` before exiting).

### Tests
`__tests__/transform-streaming-buffer-flush.spec.ts` — 5 tests: trailing `message_delta` without `\n\n`, clean-close no-duplicates, truncated JSON silent-skip, multi-byte UTF-8 flush, defer-cancel skips flush.

### Files
- `src/transform/streaming.ts` — processEvent closure + flush block
- `__tests__/transform-streaming-buffer-flush.spec.ts` — new

## 8. Gate inference features by model capability

**Commit**: `36668b8` · **Type**: bugfix · **Capability**: `model-capability-gating`

### Symptom
After a restart that brought earlier fixes into production, the user reported "sigue con los mismos putos errores" — but log analysis showed a different error: `status: 400, model: claude-sonnet-4-5-20250929, message: "adaptive thinking is not supported on this model"`. The "Connection reset by server [retrying in 28s attempt #6]" the user saw in OpenCode was the client retrying the 400 six times.

### Root cause
`src/transform/openai-to-anthropic.ts` (lines 96-101) applied three OAuth-internal inference features to every non-haiku model:
```ts
if (!isHaiku && !isStructuredOutput) {
  result.thinking = { type: "adaptive" };
  result.context_management = { edits: [{ type: "clear_thinking_20251015", keep: "all" }] };
  result.output_config = { effort: "medium" };
}
```

`thinking.type = "adaptive"` is a Claude Code OAuth feature (not in the public Anthropic SDK — public uses `{ type: "enabled", budget_tokens: N }`). Only models released around the 4-6 generation support it. Sonnet-4-5, sonnet-4, opus-4-5, opus-4-1, opus-4, opus-4-6 older versions, and haiku variants do NOT support it. Same unverified assumption applied to `context_management` and `output_config.effort`.

### Fix
New `ModelCapabilities` interface in `src/domain/models.ts`:
```ts
export interface ModelCapabilities {
  adaptiveThinking: boolean;
  contextManagement: boolean;
  outputEffort: boolean;
}
export const MODEL_CAPABILITIES: Record<string, ModelCapabilities> = {
  "claude-sonnet-4-6": { adaptiveThinking: true, contextManagement: true, outputEffort: true },
  "claude-opus-4-6":   { adaptiveThinking: true, contextManagement: true, outputEffort: true },
  // all older models explicitly false
  ...
};
const DEFAULT_CAPABILITIES: ModelCapabilities = { adaptiveThinking: false, contextManagement: false, outputEffort: false };
export function getModelCapabilities(model: string): ModelCapabilities;
```

**Decision — allowlist with safe defaults**: Unknown models default to all-false. Conservative: prefers silently losing a feature over 400-rejection.  
**Decision — investigate all 3 features together**: adaptive thinking, context_management, and output_effort all gated by the same capability map entries. Individual refinement possible later if evidence shows asymmetric support.

In `src/transform/openai-to-anthropic.ts`, the single `!isHaiku && !isStructuredOutput` block replaced with three independent capability-gated conditionals:
```ts
const caps = getModelCapabilities(model);
if (caps.adaptiveThinking && !isStructuredOutput) result.thinking = { type: "adaptive" };
if (caps.contextManagement && !isStructuredOutput) result.context_management = { ... };
if (caps.outputEffort && !isStructuredOutput) result.output_config = { effort: "medium" };
```

## 9. Forward client system prompt to model

**Commit**: `14ef8f2` · **Type**: bugfix · **Capability**: `client-system-prompt-forwarding`

### Symptom
User observed the model responded with Claude Code identity — `"Soy Claude Code, el asistente de Anthropic para desarrollo de software"` — even when the client (OpenCode) sent its own system prompt defining an OpenCode persona. User's diagnostic question: "el system prompt se manda sanitizado?" — spot on.

### Root cause
Pre-existing incomplete implementation in `src/transform/openai-to-anthropic.ts`:

- Lines 26-31 collected the client's `system` role messages into a `systemPrompt: string | null` variable.
- Lines 68-75 built the final Anthropic `system[]` array with ONLY: billing header + Claude Code identity prefix.
- `systemPrompt` was **never injected anywhere**. It was collected, logged in telemetry (`hasSystem: systemPrompt !== null`), then silently discarded.
- Constant `CONTEXT_PREAMBLE` was declared on line 10 but never used — someone started the fix and left it incomplete.

### Why not just put the client's system prompt in `system[]`?
Verified in the reference plugin `/root/opencode-claude-auth/src/transforms.ts:172-203`:
> "Anthropic's API now validates the system prompt for OAuth-authenticated requests that use Claude Code billing. Third-party system prompts (like OpenCode's) trigger a 400 'out of extra usage' rejection when they appear inside the system[] array alongside the identity prefix. Work-around: keep only the billing header and identity prefix in system[], and prepend all other system content to the first user message where it is functionally equivalent but avoids the check."

### Fix
Exported `CONTEXT_PREAMBLE`. After `firstText` extraction (so billing hash stays stable vs user intent, not the caller-provided context) and before the final `result` assembly, inject the client system prompt as a prefix on the first user message:

```ts
if (systemPrompt !== null && systemPrompt.length > 0) {
  const firstUserMsg = messages.find((m) => m.role === "user");
  if (firstUserMsg) {
    const combined = `${CONTEXT_PREAMBLE}${systemPrompt}\n\n`;
    // Three content shapes handled: string, array-with-text, array-without-text.
    if (typeof firstUserMsg.content === "string") {
      firstUserMsg.content = combined + firstUserMsg.content;
    } else if (Array.isArray(firstUserMsg.content)) {
      const blocks = firstUserMsg.content;
      const textIdx = blocks.findIndex((c) => c.type === "text");
      if (textIdx >= 0) blocks[textIdx].text = combined + (blocks[textIdx].text || "");
      else blocks.unshift({ type: "text", text: combined });
    }
  }
}
```

### Tests

- Registered for `SIGTERM` and `SIGINT`.
- Idempotent via `shuttingDown` flag.

In `/etc/systemd/system/claude-plan-api.service` (not in repo — OS-level config):
- Added `TimeoutStopSec=10` as defense in depth.

## 11. Emit SSE keep-alive heartbeats every 5s

**Commit**: `e91a66a` · **Type**: feature

### Symptom
When user asked OpenCode for "10 PDFs with reportlab", the agent repeatedly hit "Tool execution aborted" → "Connection reset by server [retrying in 6s attempt #11]". Gateway showed 200 responses with no errors. Telemetry confirmed: upstream Anthropic streamed fine, `stream.start` fired, but **13.7 seconds later** `stream.client_disconnect_deferred` with `inToolUse: true`.

### Root cause
Long `tool_use` inputs (e.g. `Write` with a multi-KB Python script as argument) produce multi-second gaps between `input_json_delta` chunks while the model writes out the argument JSON character by character. During these gaps, three candidate middleboxes drop the TCP connection:
- Tailscale / corporate NAT (idle connection cleanup)
- OpenCode's internal HTTP client inside `@ai-sdk/openai-compatible`
- Kernel-level TCP keepalive thresholds

The client retries with the same accumulated context, the model re-emits the same `tool_use` (partial JSON restart), connection drops again. From the user's side this looks like an infinite loop though each request individually returned 200.

### Fix
In `src/transform/streaming.ts`:

```ts
const KEEP_ALIVE_INTERVAL_MS = 5_000;
const KEEP_ALIVE_COMMENT = ": keep-alive\n\n";

// Inside start(controller), after reader setup:
keepAliveInterval = setInterval(() => {
  safeEnqueue(controller, KEEP_ALIVE_COMMENT);
}, KEEP_ALIVE_INTERVAL_MS);
```

SSE comment lines (leading `:`) are ignored by every conforming SSE consumer but count as TCP-level traffic, defeating idle-timeout heuristics. `safeEnqueue` already no-ops when `closed` or `pendingCancel`, so no additional guards needed.

Cleanup covers ALL exit paths:
- `finally` block at end of `start()` — covers natural end, error, break outer
- `cancel()` — covers client-cancel-early

## Architectural Decisions

### AD-1: Allowlist over denylist for model capabilities
When Anthropic adds a new model, we prefer silently losing a feature (degradation to safer defaults) over 400-rejection on a new-but-unverified model. Adding an entry when the model is validated is one line; debugging a 400 in production is hours.

### AD-2: Per-model exclusion sets do NOT reset on model switch
`getExcludedBetas(modelId)` returns per-model state that persists for process lifetime. Different models have independent exclusion sets. Rationale: if the user tried opus-4-6 and context-1m was excluded, that knowledge persists. Switching to sonnet-4-6 does not reset opus's knowledge.

### AD-3: Client system prompt travels as first-user-message prefix
OAuth-authenticated requests validate the `system[]` array for Claude Code authenticity. Third-party content inside `system[]` triggers rejections. The workaround (prepend to first user message with `CONTEXT_PREAMBLE` framing) is functionally equivalent and avoids the check. Adopted from the reference plugin.

### AD-4: Billing hash computed from original user text, not preamble-prefixed content
The billing header's CCH input is `firstText` of the first user message, extracted BEFORE system prompt injection. Rationale: billing represents user intent. Two callers with different system prompts but the same user query should get the same billing hash.

### AD-5: Tool-name mapping is richer than the reference plugin
`src/domain/tool-mapping.ts` includes semantic aliases (`task → Agent`, `todowrite → TaskCreate`, `webfetch → WebFetch`), snake_case → PascalCase auto-canonicalization, sanitization against non-alphanumeric characters, and collision handling with numeric suffixes. Not ported from plugin; pre-existing and intentionally retained.

### AD-6: Defer-cancel mid tool_use, NOT mid text
Only `content_block_start` with `type === "tool_use"` sets `inToolUse = true`. Text blocks that get interrupted mid-stream are recoverable from accumulated chunks (telemetry captures the partial text). Tool_use blocks need the complete JSON to be parseable.

### AD-7: SSE keep-alive interval = 5 seconds
Shorter than the 10-15s idle timeout observed in OpenCode's HTTP client and Tailscale middleboxes. Nginx default is 5s as well. Overhead: ~15 bytes per 5 seconds per active stream. Negligible.

### AD-8: Service lifecycle owned by systemd
Manual processes outside systemd's view cause port conflicts and confusing restart semantics. All deploys go through `systemctl restart`. The unit has `WorkingDirectory=/root/projects/claude-plan-api` so relative log paths resolve correctly.

## Lessons Learned (Gotchas)

### GT-1: `pkill -f <pattern>` can match itself
When the pattern string appears in the pkill command line, pkill matches its own process and kills the shell running it. Symptoms: the command appears to hang, output cuts mid-execution. Mitigation: use specific executable paths in the pattern, or iterate with `pgrep -f` + explicit `kill`.

### GT-2: `bun run script.ts` creates TWO processes
A wrapper and the actual script. `pgrep` returns wrapper first. Killing only the wrapper leaves the script orphaned (reparented to init), still binding sockets. Always kill ALL matching processes or use `pkill -f`.

### GT-3: Bun ignores SIGTERM by default
No implicit signal handling. Every long-running bun service needs explicit `process.on("SIGTERM", ...)` or systemd will hang on shutdown.

### GT-4: Anthropic OAuth validates system[] content
Third-party system prompts in `system[]` alongside the Claude Code identity prefix trigger 400 "out of extra usage". Workaround: keep `system[]` for billing + identity only, prepend client system to first user message.

### GT-5: `adaptive` thinking is OAuth-only
Public Anthropic SDK docs show `thinking: { type: "enabled", budget_tokens: N }`. The `adaptive` type is internal to Claude Code OAuth and only supported on 4-6+ model generations.

### GT-6: `TextDecoder` holds bytes with `{ stream: true }`
Multi-byte UTF-8 characters split across chunk boundaries are buffered inside the decoder. Always call `decoder.decode()` (no args) at end-of-stream to flush those bytes — otherwise trailing characters are silently dropped.

### GT-7: `tool_use` with orphaned pairs → model loop
Anthropic with OAuth + tools enters a degenerate state when `tool_use` blocks lack matching `tool_result` (or vice versa). The model re-emits the same tool intent repeatedly. Always sanitize pairs before dispatch.

### GT-8: Root tsconfig without `include` sweeps everything
TypeScript default-scans the full tree minus `node_modules`. Sub-projects with their own tsconfig get type-checked against the root config, producing false positives. Always declare `include` explicitly.

### GT-9: systemd `TimeoutStopSec` default is distro-dependent but always slow
Ubuntu/Debian default is 90s, RHEL-family is 90s as well. Setting `TimeoutStopSec=10` in the unit is a safe explicit value once your service has graceful shutdown.

### GT-10: `bun run` from the wrong `cwd` = logs in the wrong place
If your logger uses relative paths (`./logs/*.log`) and the process's `cwd` doesn't match the project root, logs go where `cwd` points. Always set systemd `WorkingDirectory` or launch from the correct `cwd` manually.
